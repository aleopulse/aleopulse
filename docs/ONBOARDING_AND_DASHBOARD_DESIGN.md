# LeoPulse Onboarding & Dynamic Dashboard Design

## Overview

This document defines:
1. **Onboarding Flow** - Profile selection when users first connect
2. **User Profiles** - Different user types and their needs
3. **Dynamic Dashboards** - Adaptive layouts per profile
4. **Wireframe Specifications** - Detailed screen designs

---

## Part 1: User Profile System

### Profile Types

| Profile | Primary Use Case | Key Actions |
|---------|-----------------|-------------|
| **DAO Governance** | Run governance votes for DAOs/protocols | Create proposals, manage delegation, track quorum |
| **Market Researcher** | Conduct surveys for insights | Create studies, manage panels, analyze responses |
| **HR Professional** | Gather employee feedback | Run campaigns, view org analytics, ensure anonymity |
| **Community Builder** | Engage Web3 communities | Create polls, run quests, manage rewards |
| **Survey Earner** | Earn crypto by participating | Find opportunities, build reputation, track earnings |
| **Developer** | Integrate LeoPulse into dApps | Manage API keys, configure webhooks, use SDK |

### Database Schema

```typescript
// User Profile Preferences
interface UserPreferences {
  id: number;
  walletAddress: string;

  // Onboarding status
  onboardingCompleted: boolean;
  onboardingCompletedAt?: Date;

  // Primary profile (determines default dashboard)
  primaryProfile: ProfileType;

  // Secondary profiles (can switch between)
  enabledProfiles: ProfileType[];

  // Profile-specific settings
  daoSettings?: {
    defaultSpaceId?: number;
    showDelegationWidget: boolean;
    proposalNotifications: boolean;
  };

  researchSettings?: {
    defaultStudyType?: string;
    qualityThreshold: number;
    autoScreenParticipants: boolean;
  };

  hrSettings?: {
    defaultOrgId?: number;
    anonymityLevel: 'full' | 'department' | 'team';
    minimumGroupSize: number;
  };

  communitySettings?: {
    defaultProjectId?: number;
    showQuestWidget: boolean;
    rewardNotifications: boolean;
  };

  earnerSettings?: {
    preferredCategories: string[];
    preferredTokens: string[];
    minRewardThreshold: bigint;
    showInLeaderboard: boolean;
  };

  developerSettings?: {
    defaultApiKeyId?: number;
    webhookNotifications: boolean;
  };

  // UI preferences
  theme: 'light' | 'dark' | 'system';
  compactMode: boolean;

  createdAt: Date;
  updatedAt: Date;
}

type ProfileType =
  | 'dao_governance'
  | 'market_researcher'
  | 'hr_professional'
  | 'community_builder'
  | 'survey_earner'
  | 'developer';
```

## Part 6: API Endpoints for Onboarding

```typescript
// New API routes

// Check if user needs onboarding
GET /api/user/onboarding-status/:address
Response: {
  needsOnboarding: boolean;
  partialProgress?: OnboardingState;
}

// Save onboarding progress (for multi-step)
POST /api/user/onboarding/progress
Body: {
  walletAddress: string;
  step: OnboardingStep;
  data: Partial<OnboardingState>;
}

// Complete onboarding
POST /api/user/onboarding/complete
Body: {
  walletAddress: string;
  primaryProfile: ProfileType;
  enabledProfiles: ProfileType[];
  profileSettings: Record<string, any>;
}

// Get user preferences
GET /api/user/preferences/:address
Response: UserPreferences

// Update user preferences
PUT /api/user/preferences/:address
Body: Partial<UserPreferences>

// Switch active profile
POST /api/user/switch-profile
Body: {
  walletAddress: string;
  profile: ProfileType;
}
```

## Summary

This design provides:

1. **Seamless Onboarding** - New users select their primary use case and get a personalized experience from the start

2. **Profile Switching** - Power users can switch between roles (e.g., create polls as Community Builder, earn as Survey Earner)

3. **Contextual Dashboards** - Each profile sees only relevant features, reducing cognitive load

4. **Extensibility** - New profiles can be added as new segments emerge

5. **Data-Driven** - User preferences enable personalized recommendations and analytics
